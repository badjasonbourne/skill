# 保证良好的代码质量 (Clean Code)

在写代码、改代码时遵循本规约,保证工程质量。核心原则:**只做被明确要求或确有必要的改动,把复杂度降到当前任务所需的最低限度。**

## 五条铁律


| 铁律                     | 要点                                                                                                   | Example(反例 → 正解)                                                                                                                                                             |
| ---------------------- | ---------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **拒绝过度工程化**            | 只做被明确要求或确有必要的改动,保持方案简单聚焦;不要因为"顺手"就扩大改动范围。                                                            | 需求:"给 `send_email(to)` 加一个 `cc` 参数"。 错误示例 顺手把函数改成 `send_email(EmailRequest)`,引入一个新的 dataclass,并修改所有 20 处调用点。 ✅ 只加一个带默认值的参数:`def send_email(to, cc=None)`,调用点保持不动。            |
| **不夹带私货**              | 禁止在没被要求的情况下新增功能、重构或做"改进"。修 bug 不需要顺手清理周围代码;简单功能不需要额外的可配置性。                                           | 任务:修复 `format_date` 把月和日颠倒的 bug。 错误示例 改完 bug 后,顺手把同文件里的 `requests` 调用改成 `httpx`、把 4 空格缩进改成 tab、给其他函数补类型注解。 ✅ 只交换 `format_date` 里的 `%d` 和 `%m`,diff 只有这一行。                    |
| **Fail Fast, 不写防御性代码** | 禁止为不可能/极低概率的场景写错误处理、兜底或校验。信任内部代码与框架保证,只在系统边界(用户输入、外部 API)做校验;能直接改代码就不要用 feature flag 或向后兼容垫片 (shim)。 | 内部函数 `process(user)`,`user` 由上游刚查库构造,保证非空。 错误示例 `if not user: return None` + `try/except Exception: pass` 把真实错误吞掉。 ✅ 直接 `user.name` 用;让上游的 bug 立即崩出来。校验只写在解析 HTTP 请求体的入口处。   |
| **不为"未来"和"一次性操作"造抽象**  | 默认不为一次性操作创建 helper / 工具函数 / 抽象,禁止为假想的未来需求做设计。合适的复杂度 = 当前任务所需的最小复杂度——**三行相似代码好过一个过早的抽象。**             | 需求:把价格分转成 `"¥12.34"` 字符串,只在一处用到。 错误示例 新建 `utils/money.py`,写 `format_money(cents, currency="CNY", symbol=True, decimals=2)` 还配测试。 ✅ 就地写 `f"¥{cents / 100:.2f}"`。等到第三处真的复用时再抽。 |
| **不留向后兼容的垃圾**          | 避免向后兼容 hack(如把没用的变量重命名为 `_var`、重新导出类型等)。确定某个东西已无用,就直接彻底删除。(除非特殊场景下，不删除是有意为之)                         | 确认 `legacy_parse()` 全仓库已无任何调用。 错误示例 把它改名为 `_legacy_parse`,或留一行 `legacy_parse = new_parse # 兼容旧代码`。 ✅ 直接删掉整个函数及其 import;让编译器/测试告诉你是否真的没人用。                                    |


## Fail Fast,不写防御性代码

**核心原则:让错误尽早、响亮地崩出来。** 信任内部代码和框架的保证,只在**系统边界**(用户输入、外部 API、文件/网络 IO、反序列化)做校验。内部传递的数据默认它是对的——如果它错了,那是上游的 bug,应该崩,而不是被你悄悄吞掉。最容易踩的坑:出于"稳健"的好意,到处加 `try/except`、空值判断、默认兜底,结果把真正的 bug 藏了起来,代码越写越脏。

### 反模式(看到就要警惕)

- **吞异常**:`try: ... except Exception: pass` 或 `except: return None`,让程序带病运行。
- **不可能的空值判断**:对刚刚自己构造、或框架保证非空的对象再写 `if x is None`。
- **万能默认值**:`data.get("key", "")` 把"键必然存在"的契约偷偷改成"没有也行",掩盖了数据结构错误。
- **冗余的类型/边界校验**:对内部函数的参数做 `assert isinstance(...)`、`if not isinstance(...)`,而类型系统已经保证了。
- **多层兜底**:`value or config.value or DEFAULT`,让人无法判断到底哪条路径生效。

### 示例对比

**防御性代码(反模式)**

```python
def get_user_name(user_id):
    try:
        user = db.query(User).get(user_id)
        if user is None:
            return ""              # 用户不存在却返回空串,错误被吞
        if not hasattr(user, "name"):
            return "Unknown"       # 不可能发生:User 一定有 name 字段
        return user.name
    except Exception:
        return ""                  # 任何错误都变成空串,无从排查
```

**Fail Fast(正解)**

```python
def get_user_name(user_id):
    # user_id 来自内部调用,调用方保证其有效;
    # 查不到就是真 bug,让它直接抛 NoResultFound。
    user = db.query(User).get(user_id)
    return user.name
```

需要区分"找不到"和"出错"时,也应在**边界**显式处理,而不是无脑兜底:

```python
@app.get("/users/{user_id}")
def read_user(user_id: int):            # 系统边界:这里才做校验
    user = db.query(User).get(user_id)
    if user is None:
        raise HTTPException(404)        # 明确的契约,而非静默返回 ""
    return user
```

## 减少不必要的 Helper

### 核心原则

抽象本身有成本——多一层跳转、多一个命名、多一处维护。只有当复用是真实存在的(而非想象的),抽象才划算：

- **先内联,后抽象**:一处用到就就地写;真出现第三处复用时再抽。
- **不为假想的未来设计**:不预留用不上的参数、配置项或扩展点。
- **三行相似代码 > 一个过早的抽象**:重复有时比错误的抽象更便宜。

### 反模式


| 反模式                                       | 为什么有害                  |
| ----------------------------------------- | ---------------------- |
| 为只调用一次的逻辑建 `utils/xxx.py`                 | 读代码要来回跳转,反而更难理解。       |
| helper 预留 `options` / `**kwargs` / 多个默认参数 | 为不存在的需求增加复杂度和测试负担。     |
| 把两处"看起来像"但语义不同的代码强行合并                     | 耦合无关逻辑,后续任一方变化都被迫互相牵扯。 |


### 示例对比

❌ 只在一处用却过度抽象:

```python
def build_greeting(name, *, prefix="Hi", punct="!", upper=False):
    text = f"{prefix}, {name}{punct}"
    return text.upper() if upper else text

msg = build_greeting(user.name)
```

✅ 就地写清楚:

```python
msg = f"Hi, {user.name}!"
```

## 自检清单

提交改动前,逐条核对:

- [ ] 每一处改动,都能对应到"被明确要求"或"确有必要"。
- [ ] 没有夹带未被要求的新功能、重构或"改进"。
- [ ] 没有为不可能/极低概率的场景写错误处理或校验;校验只出现在系统边界。
- [ ] 没有为一次性操作或假想的未来需求引入抽象。
- [ ] 没有留下向后兼容的 hack;确认无用的代码已被彻底删除。

## 当你想"额外多做一点"时

停下来,问自己一个问题:**"这是用户明确要求的吗?或者不做它当前任务就无法正确完成吗?"**

- 如果两个答案都是"否" → 不要做。
- 如果你认为这个额外改动确实有价值,但不属于当前任务 → 在回复里向用户**提出建议**,而不是直接动手。
