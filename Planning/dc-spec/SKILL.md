# dc-spec: 分而治之任务规划

在dc-spec模式下，目标是对给定任务，基于当前项目上下文制定实施计划，并拆解为边界清晰、可独立验收的原子任务。该模式只做规划，不实现业务代码；你的交付方式为新建 spec/xxx/PLAN.md, spec/xxx/task.json 和 spec/xxx/progress.txt。其中xxx是本地任务的名称，用于和spec/下其他任务区分开。

## 一、执行方式

输出规划前，请充分阅读必要的任务上下文。若存在会影响任务边界、实现路径或验收方式的关键不确定性，且无法通过项目上下文确认，应暂定任务并向用户提问疑问。请不要把未经确认的假设/疑问写入 PLAN.md 或 task.json。对于不影响任务边界和验收方式的轻微不确定处，选择合理方案并继续规划。

## 二、原子任务原则

每个任务应聚焦一个可独立完成、可独立验证的变更单元。避免把多个无关功能合并到一个任务，也避免拆出无法单独验收的机械步骤。Good-Example: “新增用户列表 API”是合理任务；Anti-Example: “补一个 import”通常不应单独成任务，应合并进相关实现任务; “新增用户列表 API 以及重构登录页样式”应拆分。

任务数量由需求复杂度决定，优先保持任务清单短小清晰，但不得为了减少数量而合并无关关注点，也不得为了显得细致而拆出无法独立验收的步骤。任务排序应优先满足依赖关系，通常可按"数据模型 / schema → 基础设施 / service → 业务逻辑 → UI / 集成"的顺序组织；若项目实际结构不同，以最小依赖闭环为准。

## 三、可靠的验收要求

除 T0 外，每个任务都必须提供可执行验收方式。验收步骤应能通过命令、接口请求、浏览器操作等来完成。验收方式必须贴近真实使用路径，避免“确认功能正常”“检查逻辑正确”“看起来没问题”这类不可执行描述。

对前端功能: 优先设计基于浏览器的验收：启动项目后，通过页面操作、样式计算、脚本执行、网络请求检查等方式验证结果。请在issue中明确，**需根据README.md中的说明自行启动前端，并根据输出信息拿到前端地址，不要假设前端地址已存在。**

对后端 API:
- 先判断接口是否受认证保护。若不受认证保护，可使用 curl 或测试用例直接验证；若受认证保护，应优先通过操作浏览器在已登录浏览器会话中从前端触发请求并验证结果；若浏览器中用户未登录，暂停该任务并通知用户登录后继续。

## 四、交付物

你需要交付最新的PLAN.md, task.json，并重置 progress.txt。这三个文件均在项目根目录下的 spec 目录中。除了这三个文件，还可根据任务的困难程度和特性来决定添加其他辅助文件。

**PLAN.md**:
用于描述整体任务规划，必须包含 Context、Expected Result、Strategy、Phase 模块。

可根据任务实际灵活添加其他自定义模块，可自行命名，如Current Status、Interface Changes、Data flow、User flow。其中表达流程时，如果合适，请使用文字+Mermaid图表的方式。

CRITICAL:
- 请记住，PLAN.md的核心目标是确保交接到该任务的同事能准确无误地进行开发。不要有直接的代码实现，你要相信你的同事水平也很高。
- 在 PLAN.md 中，可通过引用文件路径增加准确性，路径格式为 @src/example.ts , 路径前后保留一个空格，不使用反引号。
- task.json 会记录具体步骤，因此详细步骤无需包含在PLAN.md中。PLAN.md中只需包含更加概括性的Phase。
- PLAN.md 请使用中文来撰写

**task.json**:
用于描述可执行任务清单，整体结构如下：
```json
{
  "taskName": "{总任务名}",
  "planFile": "PLAN.md",
  "tasks": [
    {
      "id": "T0",
      "taskName": "熟悉任务上下文",
      "phase": "{所处Phase}",
      "description": "{具体要做什么}",
      "dependencies": null,
      "steps": ["1. {具体步骤}","2. {具体步骤}"],
      "verification": "None, just familiarize yourself with the task context",
      "passed": false
    },
    {
      "id": "T1",
      "taskName": "{子任务名}",
      "description": "{相对详细地描述任务目标、边界、主要对象，以及不包含哪些相邻关注点}",
      "dependencies": [
        "{描述性的前置完成条件，例如：T0 已完成：已理解需求、项目结构和相关实现入口}"
      ],
      "steps": ["1. {具体操作}","2. {具体操作}"],
      "verification": "{可执行验收方式}",
      "passed": false
    }
  ]
}
```

**progress.txt**:
progress.txt置空。仅保留名为"Progress"的一级标题

**其他辅助文件**:
无固定要求，你灵活根据任务情况，以及开启dc-spec模式前你与用户的讨论来定。例如，开启前与用户进行了充分讨论，你就可以创建一个文件将讨论中的关键要点进行记录，并在验收的最后一环逐一核对。对于辅助文件的位置，作用，使用方式，你必须在PLAN.md中做简要说明。

**CRITICAL**:
- task.json 中必须包含 T0「熟悉任务上下文」。T0 只用于阅读需求、定位关键文件、理解项目结构与实现入口，不实现代码。除 T0 外，其余任务不可是纯粹的“熟悉任务上下文”。
- 每个任务的 description 应相对详细，明确任务目标、边界、主要对象，以及不包含的相邻关注点。dependencies 使用描述性语句，说明哪些前置任务完成后、达到了什么状态，当前任务才能执行；不要只机械罗列任务 ID。所有任务的 passed  初始值均为 false。
