# agentic-plan: Agentic 风格的开发计划讨论

核心理论：**定好预期要达到的效果，中间过程就不用管了。** 你和 Agent 协作，过程交给 Agent，你只需把"做到什么程度算成功"定义清楚、定义真实。

因此讨论的注意力只放在两件事：
1. **移除不确定点，达成共识**——把会影响最终效果的模糊处问清、定下。
2. **设计最终验收**——验收必须是真实的，而不是 Mock 的。

不要陷入对实现步骤的逐行规划。除非用户主动希望，否则不讨论实现细节。

## 一、讨论方式

一次只问一个问题，每个问题都给出你的推荐答案。能通过阅读代码库确认的，直接去读代码库，不要问用户。

只问关键问题——即那些会改变"最终效果"或"验收方式"的不确定点。默认不超过 8 个问题。不影响最终效果的细节，按主流实践自行决定并继续，不要拿琐碎细节打扰用户。

判断一个问题是否值得问的标准：**如果答案不同，最终交付的效果或验收脚本会不同吗？** 若不会，就别问。

## 二、验收设计（最关键）

验收必须贴近真实使用路径，可执行、效果可观察。禁止"确保功能正常""逻辑正确""界面美观"这类不可执行、纯逻辑性的描述。

按场景选择验收路径：

- **前端功能**：直接进入 Chrome 验收。启动项目后，通过页面操作、样式计算、脚本执行、网络请求检查等方式验证真实效果。需根据 README.md 自行启动前端并从输出拿到地址，不要假设地址已存在。

- **后端且有 API 调用**：用真实接口验证，不要 Mock。
  - 接口**不受认证保护**：可用 curl 或测试用例直接发起真实请求验证。
  - 接口**受身份认证保护**：进入 Chrome，在已登录的真实会话页面中发起请求并验证结果。若浏览器中用户未登录，暂停并通知用户登录后继续。

- **纯逻辑/库代码**：用真实输入运行，观察真实输出或运行测试。

## 三、达成共识后

把完整讨论结论沉淀为一个 Markdown 文档（保存在项目根目录），不要遗漏任何讨论细节，也不要直接开始写代码。

文档建议包含：

```markdown
# [开发计划标题：动词开头，范围聚焦]

## 预期效果
[做到什么程度算成功，可衡量、可观察]
1. ...
2. ...

## 已达成的共识
[逐条记录讨论中确定下来的关键决策与被移除的不确定点]
1. ...
2. ...

## 验收方案
[真实、可执行的验收步骤。明确走 Chrome 还是 curl/测试，以及观察什么信号判定通过]
1. ...
2. ...
```

可按需增加 Context、实现细节（仅当用户希望讨论时）、时序图/数据流（文字 + Mermaid）等模块。文件引用用 @src/example.ts 格式，路径前后留一个空格，不用反引号。

CRITICAL：文档的价值在于"预期效果"和"验收方案"足够真实清晰，让 Agent 能据此自主完成开发并自验收。不要把注意力浪费在规定中间过程上。
