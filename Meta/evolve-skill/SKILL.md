# 进化 SKILL

在一次任务结束后，帮助打磨某个已有的 SKILL。读取目标 SKILL，结合刚结束的任务暴露出的问题进行反思，然后**把候选的进化方向呈现给用户挑选**。在用户认可之前，绝不修改目标 SKILL。 

## 硬性规则

1. **只提议，不自动改。** 只输出方向。在用户明确选定要改什么之前，不要碰目标 SKILL。
2. **每个方向都要有证据。** 每条方向都必须指向一个具体触发点：刚结束任务中的卡点、SKILL 的缺口、过时的指令，或用户明确提出的要求。不提"锦上添花"式的臆测改动。
3. **保持范围收敛。** 只进化用户指定的那一个 SKILL。不建议新建其它 skill，也不顺手重构无关内容。

## 工作流

### 第 1 步：定位并读取目标 SKILL

- 拿到用户指定的 SKILL。若有歧义，先问清是哪一个。
- 完整读取它的 `SKILL.md`，以及它引用的所有附属文件（reference.md、示例、脚本等）。
- 快速建立认知模型：它的用途、触发描述（description）、指令结构和隐含假设。

### 第 2 步：收集信号

从两个来源收集应当驱动这次进化的信息：

- **刚结束的任务**（主要信号）：按照 SKILL 执行时，哪里顺、哪里卡？SKILL 没能预料到什么？有哪些本该由 SKILL 告诉你、却要你自己摸索的东西？哪些指令被证明是错的、过时的或多余的？
- **用户提出的要求**（如果有）：将其作为本次进化的明确约束或目标。

如果两个来源都给不出清晰信号，就直接问用户这次进化的动因，而不是凭空编造方向。

### 第 3 步：推导进化方向

沿以下维度扫描这个 SKILL，只保留有真实证据支撑的方向：

- **新增（Add）** —— 任务暴露出的缺失步骤、知识或边界情况。
- **修正（Fix）** —— 被证明错误或过时的指令。
- **澄清（Clarify）** —— 导致犹豫或误解的含糊措辞。
- **精简（Trim）** —— 冗余、没用到或过度规定的内容。
- **触发（Trigger）** —— 当 SKILL 在错误时机触发、或该触发却没触发时，调整 `description` / 调用条件。
- **重构（Restructure）** —— 当 SKILL 不好导航时，调整顺序或渐进式披露（progressive disclosure）。
- **示例（Examples）** —— 在输出质量依赖于"看过一个例子"的地方补充具体示例。

### 第 4 步：呈现方向并等待认可

输出一个紧凑的编号清单。每条方向给出：

```
### [N] <简短标题>  · <新增|修正|澄清|精简|触发|重构|示例>
- 证据：任务 / SKILL / 用户要求中的什么触发了这条
- 拟改动：具体会改什么（位置 + 要点）
- 权衡：成本或风险（如有）
```

末尾给出明确的询问，例如："要应用哪几条？（填编号，或填'都不改'）"。然后停下等待。

### 第 5 步：只应用被认可的方向

用户选定后，只针对选中的方向修改目标 SKILL。改动保持最小、精准——不要顺手重写没被选中的段落。如果某条选中的方向实际上需要用户尚未做出的决定，再回头确认。

## 注意

- 优先选用能解决真实问题的合理改动；抵制扩张成"全面重写"的冲动。
- 如果某条方向会让 `SKILL.md` 超过约 500 行，建议把细节挪进一个一级深度的引用文件，而不是堆在主文件里。
- 用词与目标 SKILL 既有的术语保持一致。
