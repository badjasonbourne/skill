# 交互

## 可点击 Affordance

> 可交互元素必须有不依赖 hover 的常驻可见线索；用户在动手之前就该看出「这能点」。

### 为什么

用户靠控件的外形识别可交互性。一块长得像只读内容的区域，即使绑了点击事件，用户也不会去点——这是模式识别问题，不是用户的问题。只靠 `cursor: pointer`、hover 变色或底部一行灰字提示，在首屏、在触屏上都失效（触屏没有 hover）。

### 反例 → 改进

反例 —— 列表行长得和只读表格一样，仅靠 hover 变背景暗示能点：

```text
<row> 题目 A   Part 2 </row>   ← 只有移上去才变色，触屏无反馈
<row> 题目 B   Part 2 </row>
```

改进 —— 给行常驻的可交互形态：左侧 radio、右侧 chevron，或整行卡片/按钮皮肤：

```text
<row> ○ 题目 A   Part 2   → </row>   ← radio + chevron 常驻可见
<row> ○ 题目 B   Part 2   → </row>
```

### affordance 强度参考

| 线索 | 强度 | 局限 |
|------|------|------|
| `cursor: pointer` | 弱 | 要移上去才知道 |
| hover 背景变化 | 中 | 触屏无 hover；首屏不可见 |
| 辅助说明文字 | 弱 | 常被跳过 |
| 常驻图标 / chevron / radio | 强 | 一眼可见 |
| 按钮 / 卡片皮肤 | 强 | 形态本身即 affordance |

### 判断法

对每个可点元素问：「移走鼠标、换到触屏，用户还能看出它能交互吗？」 答案是否 → 加一个常驻的控件形态，而不是只加 hover 或说明文字。
